## Human Contributions

### Reviewing All Generated Code
Every file generated by an AI tool was read line-by-line before being accepted into the project. Code was not blindly committed. During review, issues such as incorrect import paths, wrong Prisma field names, mismatched response shapes, and logic that did not account for edge cases were identified and corrected.

### Modifying Business Logic
The core business logic â€” particularly balance calculation with membership date enforcement, the greedy debt simplification algorithm, and the anomaly severity-to-record-status mapping â€” was reviewed critically. Specific decisions such as "use `convertedAmount` with fallback to `amount`" and "floor outstanding balance at zero" were human judgements made after reading the generated code and considering what the application actually needed to do.

### Database Setup and Configuration
The MySQL database was created manually, the `DATABASE_URL` connection string was configured, and `npx prisma migrate dev` was run to apply migrations. Prisma Studio was used to inspect and verify the schema after migrations. These steps are entirely manual and environment-specific.

### Testing APIs
All API endpoints were tested manually using Postman and the browser network tab before they were considered working. This included:
- Verifying authentication token flow (register â†’ login â†’ use Bearer token)
- Uploading CSV files and confirming anomaly detection produced the expected results
- Checking balance and debt endpoints against manually calculated expected values
- Verifying import execution committed the correct records to the expense/settlement tables

### UI Adjustments
Generated React components were adjusted for visual consistency â€” colour values, spacing, typography weights, and responsive breakpoints were tweaked to match the project's Material UI dark theme. Component interactions (hover states, loading indicators, error messages) were tested in the browser and refined.

### Integration Work

## Verification Process

Generated code was not treated as production-ready output. The following verification steps were applied to every AI-generated file:

**1. Structural review**
The file was read top to bottom. Imports, function signatures, variable names, and exported symbols were checked for consistency with the rest of the codebase.

**2. Schema validation**
Every Prisma query field (e.g. `importSessionId`, `uploadedById`, `executionStatus`) was cross-referenced against `schema.prisma` to confirm the field names and types matched exactly. AI tools sometimes hallucinate field names that do not exist in the schema.

**3. Logic review**
The business logic in service files was traced manually â€” particularly aggregation queries, the balance fallback chain (`convertedAmount ?? amount ?? 0`), and the anomaly status classification â€” to verify the logic was correct and complete.

**4. API contract check**
Generated controller response shapes were compared against what the frontend service files expected to receive. Mismatches (e.g. a response returning `data.reports` instead of `data.report`) were corrected before connecting the frontend.

**5. Browser and Postman testing**
Every generated endpoint was called with realistic data. Responses were inspected for correct HTTP status codes, expected JSON structure, and absence of unhandled errors.

**6. Edge case testing**
Known edge cases were deliberately tested â€” empty groups (no members), import sessions with zero anomalies, CSV files with all rows invalid, groups with mixed-currency expenses and no `convertedAmount` entries.

## Limitations of AI-Generated Code

AI tools are powerful accelerators but are not infallible. The following limitations were encountered and addressed:

- **Hallucinated field names** â€” AI tools occasionally generated Prisma queries referencing fields that did not exist in the schema (e.g. `importRecord.sessionId` instead of `importRecord.importSessionId`). These were caught during structural review.

- **Outdated API patterns** â€” ChatGPT occasionally suggested patterns from older versions of libraries (e.g. MUI Grid v5 syntax in a project using MUI v9). These were corrected by checking the installed package version and current documentation.

- **Missing edge case handling** â€” Generated service functions sometimes omitted null checks or early returns for empty arrays. For example, an initial draft of the balance service did not handle the case where a group had no expenses â€” producing a divide-by-zero error in the equal split calculation. These were added after review.

- **Context drift** â€” In long AI sessions, the tool occasionally generated code that contradicted earlier decisions (e.g. using a different response field name than the one established in earlier controllers). Keeping generated code aligned with the established project conventions required active human oversight.

- **No understanding of runtime environment** â€” AI tools cannot know the state of the local MySQL database, which CSV files are available for testing, or what the actual network responses look like. All environment-specific debugging was performed manually.

In summary, AI tools significantly accelerated the scaffolding and documentation phases of this project, but the correctness, integration, and reliability of the final codebase depended on sustained human review and testing throughout development.
